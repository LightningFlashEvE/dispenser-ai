# 系统架构详解

---

## 1. 系统目标

本系统面向龙门机械臂粉末自动配料场景，采用 **Jetson Orin NX 16GB 启用 Super Mode** 作为本地 AI 主控平台，构建一套“**AI理解 + 规则校验 + 控制执行**”分层协同架构。目标是在不依赖外网的前提下，实现语音交互、视觉识别、任务结构化、受控执行与全过程可追溯。

---

## 2. 系统边界与职责分工

### 2.1 Jetson 主控职责

- 本地 ASR / LLM / TTS
- 固定工位视觉检测、二维码识别、坐标换算
- 网页前后端服务
- 药品、配方、日志、异常信息管理
- 规则引擎与状态机运行
- 与后级控制程序的任务通信

### 2.2 后级控制程序职责

- 机械臂动作执行
- IO 联锁
- 称重闭环
- 急停与限位处理
- 超时与异常停机
- 执行结果状态反馈

### 2.3 关键边界原则

1. AIExtractor 只输出本轮字段 patch，不判断任务完整性
2. AI 不生成 command JSON，不直接生成底层运动控制指令
3. AI 不能写 `chemical_id` 或 `slot_id` / `motor_id` / `pump_id` / `valve_id` / `station_id` 等控制层字段
4. `chemical_id` 只能来自 catalog lookup 或用户候选选择
5. 用户确认 = 本机自审批，但必须先通过后端规则校验
6. 前端不直接控制设备
7. 任务必须经过规则引擎、状态机与白名单校验

---

## 3. 分层架构

系统采用五层结构：

1. **交互层**：语音输入/输出、网页触摸界面
2. **AI 处理层**：ASR、LLM、TTS
3. **视觉层**：固定工位检测、二维码识别、位置标定
4. **控制适配层**：任务 JSON 校验、规则引擎、状态机、协议适配
5. **执行与数据层**：后级控制程序、机械臂、称重模块、SQLite 数据库、日志系统

## 5. 核心执行链路

**用户输入 → Intent Router → AIExtractor patch → DraftManager merge → Validator → ASR guard → Chemical Catalog lookup → 前端结构化确认卡片 → 用户确认 self-approved → 后端规则校验 → 状态机检查 → 生成 command JSON → C++ 后级控制程序 → 机械臂/下料电机执行 → 执行反馈 → 界面与语音反馈**

天平数据流（并行，仅展示）：
**天平 → RS422-USB → Jetson 后端（MT-SICS 驱动）→ mg 整数 → WebSocket 推前端实时展示**
（称重闭环和停止时机由 C++ 控制程序自主决定，AI 层不干涉）

说明：
- AIExtractor 只输出字段 patch；后端 DraftManager 合并草稿，Validator 判断缺失字段和是否 `READY_FOR_REVIEW`
- ASR guard 保留 `raw_text` / `normalized_text` / `confidence`，低置信关键字段必须先确认
- Chemical Catalog lookup 负责确认真实化学品身份；`chemical_id` 不来自 AI
- 用户确认卡片后，后端生成 proposal / intent JSON，并标记 `approval_mode=self_approved`
- 规则引擎校验后生成 `command JSON`（格式见 `shared/command_schema.json`），为执行层，字段值来自数据库和已确认 catalog 数据
- C++ 后级控制程序负责把 command JSON 映射为具体机械动作
- 未通过规则校验的任务不得生成 command，更不得下发

## 6. 通信分层

### 6.1 前端 ↔ 后端
- 协议：`HTTP/REST + WebSocket`
- 用途：页面交互、状态订阅、任务结果展示、日志与药品信息管理、**实时天平读数推送**

### 6.2 后端 ↔ C++ 后级控制程序
- 协议：`TCP + JSON`（命令下发和状态回调）
- 用途：command JSON 下发、同步接收确认、异步执行结果回调

### 6.3 后端 ↔ 天平（WKC204C）
- 协议：`RS422-USB + MT-SICS ASCII 串口协议`
- 用途：实时读取重量，转换为 mg 整数；执行期间监控是否达到目标值
- **天平直接连接 Jetson，不经过 C++ 控制程序**

### 6.4 C++ 后级控制程序 ↔ 执行设备
- 电机驱动器：`RS485 + Modbus RTU`（或按实际驱动器协议）
- 其他现场设备：`CAN / 串口 / TCP`（按实际设备确认）

## 7. 模块设计

### 7.1 语音 / draft 链路
- silero-vad：语音起止检测
- whisper.cpp：中文离线识别
- ASR guard：保留 raw / normalized / confidence，阻止低置信关键字段静默进入确认态
- Intent Router：保守分流普通对话、查询、任务草稿、取消和确认
- 本地 LLM（llama.cpp server，Qwen3-4B-Instruct-2507-Q4_K_M）：只做语言理解和字段 patch 提取
- DraftManager：合并 session 级 draft、持久化、记录审计事件
- Validator：判断 missing_slots、pending_confirmation_fields、READY_FOR_REVIEW
- Chemical Catalog lookup：匹配真实化学品 ID、CAS、等级；多候选必须用户选择
- 规则引擎：校验 proposal / intent、库存、设备状态和安全边界，生成 `command JSON`（格式见 `shared/command_schema.json`）
- MeloTTS：语音反馈和反问播报

### 7.1.1 天平驱动链路（常驻）
- pyserial：读取 RS422-USB 串口，发送 MT-SICS 命令（`S`/`SI`/`Z`）
- 驱动层：解析响应，转换为 mg 整数
- 数据分发：实时 WebSocket 推送前端展示
- **职责边界**：天平接 Jetson 仅用于**前端实时展示读数**，称重闭环（何时停止下料）由 **C++ 控制程序自主决定**，AI 层不干涉

### 7.2 视觉链路
- 固定 ROI 工位有无检测
- QRCodeDetector 解析二维码
- 标定关系换算工位坐标/抓取参考位置
- 识别失败时允许进入半自动或人工介入模式

### 7.3 控制适配层
- 参数完整性检查
- 数值范围校验
- 当前设备状态确认
- 安全联锁检查
- 执行结果回读
- 超时处理与失败回滚

### 7.4 运行态调度策略

#### 服务分类

| 分类 | 服务 | 说明 |
|------|------|------|
| 常驻 | 天平驱动（MT-SICS over RS422-USB） | 持续读取重量，推送前端，执行期间监控目标值 |
| 常驻 | 视觉（固定ROI+QRCodeDetector） | 工位有无检测、二维码识别，持续运行 |
| 常驻 | C++ 控制程序通信（TCP） | 任务下发与状态回读 |
| 常驻 | Web 服务 | 网页界面、状态显示 |
| 常驻 | 控制适配层 | 规则引擎、状态机、协议适配 |
| 懒加载 | ASR（whisper.cpp） | 按需启动，语音识别完成后释放 |
| 懒加载 | TTS（MeloTTS） | 按需启动，播报完成后释放 |
| 常驻 | LLM（llama.cpp server） | 常驻运行于 8080 端口，GPU 全层推理 |

#### 优先级定义（高 → 低）

| 优先级 | 服务 |
|--------|------|
| P0（最高）| 后级控制通信与状态回读 |
| P1 | 视觉工位检测 |
| P2 | 网页状态显示 |
| P3 | ASR 语音识别 |
| P4 | TTS 语音合成 |
| P5 | LLM 深度推理 |
| P6（最低）| 日志快照落盘 |

#### GPU 资源分配
- LLM 深度推理为 **GPU 独占** 服务
- 其他服务（视觉、ASR、TTS）不占用 GPU 资源

#### 高负载降级顺序
日志快照落盘 → TTS → ASR → LLM 深度推理

#### 互斥关系
- LLM 与其他 GPU 服务互斥
- ASR 与 TTS 互斥（避免回声）

## 8. 数据与日志

建议使用以下本地存储：

- `SQLite`：药品、配方、工位、任务、执行记录、报警记录等结构清晰、需要查询统计和关联分析的业务数据
- `JSON`：系统配置、标定参数、模型配置、规则模板、LLM 原始输出快照等层级较深、人工可读性要求较高的数据

本系统采用“**SQLite + JSON 文件**”的混合存储架构。其中，SQLite 负责业务数据的规范化存储与可追溯查询，JSON 负责配置类和快照类数据的灵活管理与人工维护。高频实时控制状态原则上以内存态管理，必要时再按规则落库或落盘。该方案兼顾了 Jetson 边缘设备本地部署场景下的低运维成本、数据可追溯性、配置可维护性和故障复盘能力，适合离线工业控制应用。

日志建议覆盖：
- 语音识别结果
- AIExtractor raw output、sanitized patch、applied patch
- DraftManager audit events
- ASR raw_text / normalized_text / confidence / needs_confirmation
- Catalog lookup 候选和用户选择
- 规则校验结果
- 任务下发记录
- 控制执行反馈
- 异常与降级事件

## 9. 降级运行模式

### 全功能模式
- 语音 + 视觉 + 自动配料 + 日志 + 网页界面

### 半自动模式
- 视觉识别 + 人工确认 + 自动执行

### 手动模式
- 通过网页按钮直接驱动后级控制程序，不使用语音与 LLM

### 维护模式
- 相机标定、日志导出、设备诊断、参数调整

## 10. 关键设计原则

1. **本地优先**：核心能力完全本地化
2. **AI 只填草稿**：AI 只提取 patch，不判断完整性，不生成 command，不写化学品 ID 或控制层字段
3. **规则隔离**：用户确认是本机自审批，但执行前必须经过规则与状态机
4. **控制隔离**：后级控制程序负责设备级动作
5. **安全优先**：联锁、急停、异常处理优先于任务执行
6. **可追溯**：全过程记录日志与反馈
